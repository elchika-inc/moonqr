# Contributing to moonqr

Thanks for your interest in contributing. This project has a couple of build-order and
tooling quirks that are easy to trip over on a first checkout — read this before you dive in.

## Prerequisites

- **MoonBit toolchain** (compiles `core/` to JavaScript). This repo is verified against
  **`0.1.20260703` (`6fbf8c3` 2026-07-03)** — installing `latest` may give you a newer,
  incompatible toolchain. See [`docs/TOOLCHAIN.md`](docs/TOOLCHAIN.md) for why the version
  can't actually be pinned by the official installer, and what to do about it:

  ```sh
  curl -fsSL https://cli.moonbitlang.com/install/unix.sh | bash
  export PATH="$HOME/.moon/bin:$PATH"
  moon version   # compare against docs/TOOLCHAIN.md
  ```

- **Node.js 22+** (CI runs on Node 22; the published npm packages declare `engines.node
  >=18.18.0`, but use 22 locally to match CI).
- **pnpm 10** (`corepack enable` or `npm i -g pnpm@10`).

```sh
pnpm install --frozen-lockfile
```

## Build order matters

The TS packages import MoonBit's compiled JS output directly from `core/_build/` (note:
`_build/`, **not** `target/`), e.g. `packages/moonqr/src/core-decode.ts` imports from
`../../../core/_build/js/release/build/decode/decode.js`. That directory only exists after you
build `core/` — building the TS workspace first will fail (or silently pick up stale output).
Always build `core/` before `pnpm -r build`:

```sh
# 1. MoonBit core — must run first
cd core && moon test --target js && moon build --target js --release && cd ..

# 2. TS packages — tsup inlines core's JS output from core/_build/
pnpm -r build
pnpm -r typecheck
```

`.github/workflows/ci.yml` follows this exact order; if you change it there, mirror the change
here.

## The four test layers

Each layer proves something different — run all four before opening a PR.

1. **`moon test --target js`** (run from `core/`, 98 tests) — MoonBit unit tests for the
   encode/decode algorithms themselves (bit packing, Reed–Solomon, binarization, finder-pattern
   location, mask scoring, etc.), independent of the JS/TS wrapper layer.
2. **`node --test` in `packages/moonqr`** (273 tests) — end-to-end correctness against two
   external references:
   - **Decoder parity**: every ground-truth-decodable case in jsQR's own `tests/end-to-end/`
     corpus (214/214) must decode to the exact expected text.
   - **Encoder sweep**: a 160-combination sweep (all 40 versions x 4 EC levels) compared
     bit-for-bit against the `qrcode` npm package's module matrix.

   **Requires fetching fixtures first** — they are not checked in:

   ```sh
   node scripts/fetch-fixtures.mjs
   node --test packages/moonqr/test/*.test.mjs
   ```

   `fixtures/` is gitignored; the script shallow-clones jsQR's `tests/end-to-end/` at a
   **pinned commit SHA** (see "Code provenance" below) into `fixtures/jsqr-e2e/`. It's
   idempotent — it skips re-fetching if the expected case count is already present.
3. **`pnpm --filter @elchika-inc/moonqr test:unit`** (vitest, 44 tests) — wrapper-level
   behavior for the `moonqr` package (subpath exports, `/dom` canvas rendering, error handling
   at the JS boundary).
4. **`pnpm --filter @elchika-inc/moonqr-scanner test:unit`** (vitest, 24 tests) — `QrScanner`
   behavior (camera lifecycle, worker fallback, multiscale retry), run against a DOM shim.

## Generated files that are gitignored but required

- **`packages/scanner/src/worker-inline.generated.ts`** — the scanner's Web Worker is bundled
  to a self-contained IIFE string and inlined here so it can be started from a `Blob` URL (blob
  origins can't resolve `node_modules`). It's regenerated automatically by the package's
  `typecheck` and `build` scripts (`node scripts/build-worker.mjs`); you don't need to run it
  by hand, but don't be surprised that it doesn't exist on a fresh checkout until you do.
- **`core/src/encode/tables.mbt`** and **`core/src/decode/sjis.mbt`** — script-generated (see
  below), gitignored *source* is not the right mental model here: these two files **are**
  checked in, but are generated by `scripts/gen-tables.mjs` / `scripts/gen-sjis.mjs`. Don't
  hand-edit them; regenerate and diff instead.

## Code provenance rules

- The decoder is a **faithful port of [jsQR](https://github.com/cozmo/jsQR)** (Apache-2.0) at a
  pinned upstream commit (`8e6a036beafa7053dd44b1b76ac578d22b1b3311`). If you change a ported
  algorithm (binarization, locator, extractor, Reed–Solomon decoding, format/version info,
  segment/data decoding), **explain why in the PR description** — a silent behavioral change
  here is much more likely to be a bug than an improvement — and re-run the jsQR corpus parity
  test (layer 2 above) to confirm you haven't regressed the 214/214 match.
- `core/src/encode/tables.mbt` (Reed–Solomon block table, alignment-pattern positions) is
  generated from [qrcode-generator](https://github.com/kazuhikoarase/qrcode-generator) (MIT) by
  `scripts/gen-tables.mjs`, and `core/src/decode/sjis.mbt` (Shift-JIS code table) is generated
  from jsQR's `shiftJISTable.ts` by `scripts/gen-sjis.mjs`. Both scripts pin an upstream commit
  in a header comment. Don't hand-edit the generated `.mbt` files — change the generator script
  and regenerate, then `git diff` the result to confirm the change is what you expect.

## PR expectations

- All four test layers green (see above), plus `pnpm -r typecheck`.
- **Do not run `moon fmt`.** On some toolchain versions it triggers a repo-wide MoonBit config
  migration that touches unrelated files; format MoonBit code by hand to match the surrounding
  style instead.
- Keep changes scoped — this is a small, algorithm-heavy codebase where unrelated reformatting
  makes ported/generated code harder to diff against upstream.
